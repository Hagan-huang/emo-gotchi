# 專案交接文件：小怪獸心情日記 (Frontend Status & Backend Requirements)

這份文件旨在協助第二位後端負責人（以及協作的 AI）快速了解目前的專案進度、前端已實作的功能細節，以及進入後端開發初期所需的架構配合與開發建議。

---

## 第一部分：前端現有功能與細節 (Frontend Specifications)

目前前端已完成單一頁面應用程式 (SPA) 的開發，主要著重於「設計系統的統一」、「組件模組化」、「流暢動畫與過場」以及「友善的邊界狀態處理」。以下是各個主要視圖與功能細節：

### 1. 核心設計系統與共用組件
* **視覺與排版**：統一使用溫暖大地色系（如深咖啡 `#5D4037`、米黃 `#FFFAEC` 等），搭配粗邊框、圓角與陰影，營造出繪本風格。
* **動畫與微交互**：大量使用 CSS 關鍵影格與 Tailwind 的 transition（元素淡入 fade-in、點擊彈跳/微縮放、全螢幕粒子的呼吸燈背景等）。
* **UI 組件庫**：
  * **ConfirmModal (確認彈窗)**：統一樣式的警告與確認彈窗。已使用 `React Portal` 將其掛載到 `document.body`，避免 Z-Index 或容器 `overflow: hidden` 造成的截斷問題。
  * **TutorialOverlay (新手引導教學)**：透過 Portal 覆蓋全螢幕的半透明提示框，已串接 `localStorage` 防止重複顯示。
  * **Toast (提示通知)**：全域推播通知，用於顯示對話成功、任務完成或錯誤提示。

### 2. 首頁 (HomeView) - 互動與養成機制
* **狀態預覽**：怪獸的立繪 (MonsterAvatar) 會根據孵化狀態（蛋或怪獸）、目前情緒（生氣、憂鬱、開心等）、配備的飾品動態渲染。
* **互動體驗**：
  * 點擊小怪獸會觸發音效與愛心漂浮動畫。
  * 對話框：顯示小怪獸的隨機心聲；用戶可在輸入框輸入當局心情。
  * 隨機掉落：對話後具備模擬的隨機機率觸發「獲得配件」彈窗。
* **孵化機制**：一開始怪獸是「蛋」的型態，透過對話或任務完成次數累積，前端模擬了破殼而出的過程。

### 3. 任務系統 (TasksView)
* **狀態判斷**：當怪獸還在蛋裡時，介面會呈現 Empty State 提醒用戶「還沒孵化喔！先去首頁說說話」，限制進行任務。
* **列表與重置**：每日任務列有完成/未完成狀態，點擊完成會觸發 Toast 通知以及畫面上方的「倒數計時」提示距離午夜刷新還有多少時間。

### 4. 心情日記與圖鑑 (DiaryView)
* **圖鑑檢視區**：左側為過往下方的側邊欄列表（在手機板可以收起/展開），記錄歷史小怪獸的狀態與養成天數。
* **資料視覺化**：右側詳細記錄使用了 `Recharts` 繪製環形圓餅圖，顯示該隻怪獸成長期間經歷的各種心情分佈比例。
* **放生防呆系統**：支援「放生怪獸」將當前小怪獸封存入日記。觸發放生時會彈出 `ConfirmModal` 進行防呆確認，若還是蛋則會拒絕放生動作。

### 5. 社群分享與照片牆 (CommunityView)
* **探索與分類**：支援切換「探索社群」與「我的貼文」。在拉取資料時實作了 Spinner (Loader) 以及空集合的 Empty State UI。
* **卡片與彈窗互動**：貼文有自訂是否公開「養育天數」與「故事」。點開貼文會有詳細的放大檢視 Modal，並針對底部互動區（按讚、留言、分享）加強了白色邊框底板以利與背景區隔。
* **發布貼文**：能夠從「當前小怪獸」或「圖鑑中歷史怪獸」擇一發布貼文。

## 第二部分：後端開發建議與配合清單 (Backend Requirements Definition)

目前前端資料多仰賴 `/src/api/mock.ts` 與 Context API 的本地狀態管理。為了讓小怪獸能夠跨裝置登入並且能與真正的 AI 對接，後端需要配合處理以下項目：

### 1. 用戶與認證系統 (Authentication / Account System)
* **帳號建立**：設計會員註冊/登入 (OAuth / JWT)，確保每個使用者的資料獨立。
* **資料關聯**：後續所有操作（當前怪獸狀態、圖鑑、貼文、任務進度），都需與特定的 `user_id` 強關聯。

### 2. 小怪獸狀態機與持久化 (Monster State Persistence)
* **資料庫設計**：參照前端 `types.ts` 中的 `MonsterState` 設計欄位，例如：`isEgg` (是否為蛋), `hatchTime` (孵化時間戳), `emotionLabel` (情緒狀態), `conversationCount` (對話次數) 等。
* **後端驗證 (Source of Truth)**：
  * 用戶的配件 (Accessories) 獲取與裝備狀態需由後端保存。
  * 小怪獸成長的相關數值運算（例如與 LLM 聊完天心情分數加多少、累計多少次可以孵化），**必須**由後端進行邏輯核算後吐給前端，不應信任前端傳來的值以防作弊。
  * 「放生」API：需在後端阻擋「蛋」階段的放生請求，放生後應在資料庫封存該紀錄（轉入 Diary 表），並生成一個新的蛋狀態給該用戶。

### 3. AI 大語言模型對接 (LLM / AI Integration)
* **對話 API `/api/chat`**：
  * 前端會送出用戶的心情敘述。後端需將該敘述餵給 LLM (例如 OpenAI / Gemini)。
  * **Prompt 設計建議**：需請 LLM 根據用戶輸入，判斷這句話的正負面情緒，並回傳格式化 JSON (如：怪獸回應台詞、情緒變化值、本次想表現的情緒 Label)。
  * 後端根據 LLM 回傳的數據，同步更新資料庫中的怪獸數值，再將結果回傳給前端。

### 4. 每日任務引擎 (Daily Quests Engine)
* **時區與重置邏輯**：前端目前只能算本地時間，後端需接管「任務每日凌晨重置」的邏輯 (`/api/tasks`)，建議記錄任務上次更新的時間戳，當跨日時由後端發送新的任務列表清單。
* **任務獎勵結算**：用戶點擊完成任務發送 Request，後端需驗證是否真正滿足條件並給予資源反饋。

### 5. 社群照片牆與互動 (Community Features)
* **發布貼文 `/api/community/posts`**：
  * 前端會傳入 `monster_id` (來自當前或日記)、`story` (故事內容) 及隱私選項 (`showStory`, `showDaysOld`)。
  * 後端需將此紀錄獨立存入貼文表，並帶有當下怪獸外觀的快照。
* **留言與按讚機制**：
  * 支援 `/api/community/like` 與 `/api/community/comment`。
  * 針對公開貼文探索，需要實作**分頁 (Pagination)** 或**無限滾動機制**的拉取方式以保證效能，並支援依照上架時間或按讚數排序。
  * 備註：初期可不嚴格防洗版，但中後期應加入 Rate Limiting 機制。

### 6. Loading 與 Error 狀態協作
* **全域錯誤碼約定**：後端若遇到驗證錯誤（如：權限不足、狀態不對等），請傳遞清晰的 HTTP Status Code (400/401/403/500) 與 `message`。前端已設計了全局的 Toast 與 Loading 元件，可直接捕獲錯誤並優雅地反饋給用戶。
